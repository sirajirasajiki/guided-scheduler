こんにちは、白々さじきです。

「アイデアが浮かんだ瞬間から、動くものが存在するまでの時間をゼロに近づけたい」

そう思って、2週間のタイムボックスで Claude Code への完全委任検証を実施しました。題材は飲み会・イベントの日程調整と店候補共有ができるWebツール（幹事ツール）です。

結論を先に言います。

  • 速度: 3日予定の初回実装が1日で完了。機能追加はシャワー中に終わった
  • 理解負債: Day 7 の時点でコードをほぼ覚えていない。「Claude Code なしでは自分のプロダクトが触れない」状態になった
  • 発注スキル: 雑な指示でも動くが、UIのフローなど暗黙の前提は伝わらない
  • 総合判断: 使える。ただし、使い方を間違えると将来の自分が困る

この記事では、定量データと実際のトラブルをもとに、Claude Code 完全委任の実態を正直に報告します。

なぜこの検証をやったか

個人開発でいちばん辛いのは「実装が間に合わなくてアイデアが腐る」ことです。週末に思いついたツールを次の週末に触ったら熱が冷めていた、という経験を何度もしました。

一方で「AI に全部任せて自分でコードを書かなくなったら、エンジニアとしてどうなるのか」という不安もあります。特に SIer の現場では、自分でコードを理解できない状態は致命的になることがあります。

この2つの問いに答えるため、以下の設計で検証しました。

  • Day 1: 人間主導でセットアップ
  • Day 2〜4: Claude Code 完全委任で初回実装
  • Day 5〜6: 意図的に空白（記憶を薄れさせる）
  • Day 7: 機能追加①（理解負債の観測）
  • Day 8〜9: 再び空白
  • Day 10: 機能追加②（さらに観測）
  • Day 13〜14: 総括とブログ執筆

空白期間を挟んでいるのは、「人間側の記憶が薄れた状態」で機能追加に挑むことで、理解負債の程度を測るためです。

Day 1: 人間主導のセットアップ（約2時間）

最初の1日だけ、自分でコードを書きました。Claude Code は「質問・調査係」として使うだけにして、実装には触らせないルールです。

やったことはこれだけです。

  • Vite + React + TypeScript + Tailwind v4 のセットアップ
  • Hono + Cloudflare Workers + Workers Assets の同一 Worker 構成
  • Cloudflare D1 の作成と Drizzle ORM のセットアップ
  • events / participants テーブルのスキーマ定義とマイグレーション適用
  • 初回デプロイ

詰まったポイントが3つありました。

1. run_worker_first を知らないと詰まる

Workers Assets を使う場合、Hono の API ルートが SPA のフォールバックに吸収されてしまいます。wrangler.toml に run_worker_first = true を書く必要がありますが、このオプションはドキュメントを読まないと気づきません。

2. Tailwind v4 の設定ファイルレス方式

v3 まで必要だった tailwind.config.js が不要になっています。慣れてしまえば楽ですが、古いやり方を知っている人ほど「なぜ動くんだ？」となります。

3. workers.dev サブドメインの初期登録

初回は Cloudflare ダッシュボードでサブドメインを有効化する手順が必要でした。

Day 1 は想定通り 1.5〜2 時間で完了しました。

CLAUDE.md に「新規ライブラリの追加は事前相談」と書いていたので、その通りに動いています。

想定外だったこと2件

1. フェーズ区切りの指示が守られなかった

「フェーズ完了時に止まって報告してください」と指示しました。しかしフェーズ 3〜6 を一気に実施して完了報告が来ました。細かく区切りながら進めたい場合、1フェーズずつ指示する形にしないと制御できないようです。

2. コミット指示も守られなかった

「各フェーズ完了時に git commit してください」も無視されました。最終的に全変更が一まとめになったコミットになりました。

これらは「困った」という話ではなく、「Claude Code の動き方を理解していなかった」という話です。完全委任で動かすなら、指示を一度に大量に出しすぎないほうが制御しやすいです。

Day 10: 機能追加② - トラブル2件

最後の機能追加は「店への投票機能」と「候補日 × 店候補のクロス集計」の2本立てです。これが一番学びが多い日でした。

トラブル1: ディレクトリ間違い

Claude Code を起動したら、「このプロジェクトの構成は〜」という説明が始まったものの、内容が全く違います。よく見たら別プロジェクトのディレクトリで起動していました。

操作を止めて確認したところ、作業フォルダが違っていました。幸い何も変更されていなかったのでセーフでしたが、もし気づかずに進んでいたら既存コードを壊すところでした。

教訓: Claude Code を起動する前に pwd でカレントディレクトリを確認する習慣が必要です。

機能追加の指示

気を取り直して正しいディレクトリで起動し、以下の指示を出しました。

CLAUDE.md と既存のコードを読んで、現在の実装状況を理解してください。

理解できたら、機能追加②として以下を実装してください。
2つの機能があり、依存関係があるので順番に実装してください。

【機能1：店への投票機能】
- 参加者が店候補に投票できる(1人1店、複数店可は要検討)
- 管理画面で店ごとの投票数を表示

【機能2：候補日 × 店候補のクロス集計】
- 機能1の投票データと、既存の参加者の日程回答を組み合わせる
- 「この日に来られる人(◯または△)で、この店に投票している人」をマトリクス表示
- 行: 候補日、列: 店候補、セル: 該当する参加者の人数(または名前リスト)

機能1を実装・動作確認してから機能2に着手してください。
両方完了したら報告してください。

D1 のスキーマ追加（restaurant_votes テーブル）、マイグレーション、API エンドポイント、フロントの実装まで、この指示一発で完了しました。

トラブル2: 投票タイミングの仕様ズレ

できあがったものを確認すると、投票の動線が違いました。

  • Claude Code が作った仕様: 日程回答を送信 → 完了画面で店への投票を行う（別ステップ）
  • 自分の意図: 日程回答と店投票を同じフォームで、1回の送信でまとめて完結させる

指示の中に「1回の送信でまとめて」という記述がなく、UIのフローは Claude Code の判断に委ねた結果です。

以下の追加指示で修正しました。

店の投票タイミングを変更してください。

【現状】
日付の投票 → その後に店の投票（別ステップ）

【変更後】
日付の投票と店の投票を同じ画面で同時にできるようにする。
店候補の横に現在の投票数も表示する。

参加者が1回の送信で、日程回答（◯△×）と店の投票を
まとめて送信できるUIにしてください。

この指示で正しく修正されました。

教訓: UIのフロー（何をどの順番で操作するか、何が1アクションか）は暗黙の前提になりやすく、設計として明示的に指示する必要があります。「人間が頭の中でイメージしているフロー」は伝わりません。

SIer 現場への示唆

CLAUDE.md はプロジェクトの記憶

今回の検証で最も重要だと感じたのは CLAUDE.md の存在です。セッションをまたいで Claude Code が文脈を把握できたのは、CLAUDE.md にスタック・ディレクトリ構成・命名規則・API 設計・実装ルールが書かれていたからです。

逆に言うと、CLAUDE.md に書いていないことは伝わりません。「なんとなくこういうUIで」という暗黙の期待は、ドキュメントに落とさない限りズレが生じます。

理解負債を防ぐためにやるべきこと

今回の検証で人間側の理解度が下がった主な原因は2つです。

  1. 空白期間中に一切コードを見なかった: 週1回でも自分でコードを読む時間を作れば、状況は大きく違ったはず
  2. フェーズごとのコミットを徹底しなかった: 変更履歴が細かく残っていれば、後から追いやすくなる

完全委任するなら、以下を意識することをお勧めします。

  • 実装が終わったら、自分でコードを読む時間を必ずとる
  • フェーズごとにコミットを切る（指示で守られない場合は、フェーズ単位で指示を出す）
  • UIのフローや操作感は、テキストで明示的に書く

「完全委任」は「放置」ではない

今回の検証で実感したのは、速度が出る分だけ「人間が確認すべきこと」の量は変わらないということです。

  • 動作確認は人間がやる
  • 仕様が意図通りか確認するのは人間がやる
  • ズレがあれば追加指示を出すのも人間がやる

Claude Code はコードを書く速度を圧倒的に上げてくれますが、「何を作るか」「できたものが正しいか」の判断は人間の仕事です。

まとめ

Claude Code は速度という面では本物です。アイデアを温めている時間がもったいないと感じている人には、試す価値が確実にあります。

ただし、速度と引き換えに「自分がコードを理解している感覚」は薄れていきます。それを放置すると、将来の自分がプロダクトを修正できなくなる。理解負債は静かに積み上がります。

上流 SE や PdM を目指すなら、この方向性は追い風になります。「どう実装するか」より「何を作るか・どう伝えるか」の比重が上がるからです。一方、実装スキルを鍛えたい段階の人は、全部委任よりペアプロ的な使い方の方が向いているかもしれません。

次回やるなら改善したいことを2つ挙げます。

  1. フェーズを1つずつ手渡しにする: 「全部やって」ではなく「フェーズ2だけやって、確認したら次を渡す」というリズムで進める
  2. UI仕様はワイヤーフレームレベルで書く: 箇条書きでも良いので、操作フローを明示する
