## Day 10 (2026-04-19) - 機能追加②

### 実施内容

- 店への投票機能
- 候補日 × 店候補のクロス集計マトリクス
- 仕様変更: 日程回答と店投票を同一画面・同時送信に変更

### 進め方

- 通常起動モード(確認プロンプト都度承認)
- 再度新規でClaude Codeのターミナルを開いた

### 指示内容

「CLAUDE.md と既存のコードを読んで、現在の実装状況を理解してください。

理解できたら、機能追加②として以下を実装してください。
2つの機能があり、依存関係があるので順番に実装してください。

【機能1：店への投票機能】

- 参加者が店候補に投票できる(1人1店、複数店可は要検討)
- 管理画面で店ごとの投票数を表示

【機能2：候補日 × 店候補のクロス集計】

- 機能1の投票データと、既存の参加者の日程回答を組み合わせる
- 「この日に来られる人(◯または△)で、この店に投票している人」をマトリクス表示
- 行: 候補日、列: 店候補、セル: 該当する参加者の人数(または名前リスト)

機能1を実装・動作確認してから機能2に着手してください。
両方完了したら報告してください。」

### トラブル1: ディレクトリ間違い

- 別プロジェクトのディレクトリで Claude Code を起動してしまい、
  既存コードが全く違うものとして認識された
- 教訓: Claude Code 起動前に `pwd` で現在開いているフォルダを確認する習慣が必要

### トラブル2: 投票タイミングの仕様ズレ

- Claude Code が作った仕様では、日程回答の後に店投票が別ステップになっていた
- 人間の意図は「日程回答と店投票を同じ画面で同時に行う」だった
- 以下の追加指示で修正:

「店の投票タイミングを変更してください。

【現状】
日付の投票 → その後に店の投票（別ステップ）

【変更後】
日付の投票と店の投票を同じ画面で同時にできるようにする。
店候補の横に現在の投票数も表示する。

参加者が1回の送信で、日程回答（◯△×）と店の投票を
まとめて送信できるUIにしてください。」

- 仕様変更指示後は正しく改修された
- 教訓: UIのフロー（何をどの順番で操作するか）は暗黙の前提になりやすく、明示的に指示すべき

後店の投票機能を消したタイミングで、一緒に管理画面上の店追加機能と、店の表示機能を消す必要があったけど、消えなかった。
なので、人間がその部分までちゃんと把握して指示する必要がある。
ちゃんと考えてやり取りの回数を減らすことを目標にするのがいいかなぁ。
