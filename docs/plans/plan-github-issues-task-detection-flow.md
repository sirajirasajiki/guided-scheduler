### ## 委任フローの全体像
- 発注側（人間）がやること：方向性の指示だけ
- Claude Code が担う：Issues確認 → バグ選定 → 計画立案 → 実装
- 図示（テキストベースのフロー図）

### ## 実際のプロンプト例
- 「バグのラベルのものを見せて」で一覧確認
- 「その中から取り組みやすそうなものを選んで」で選定委任
- → Issue #14「← トップに戻る」が対象になった流れを紹介

### ## なぜこれが「発注スキル」なのか
- 従来の委任イメージ：「◯◯を実装して」と具体的な仕様を渡す
- 今回の発見：「何をやるか」の選定自体を委任できる
- 前提：Issues が整備されていること・ラベルが機能していること
- 教訓①：**Issues の粒度・ラベル整備が委任品質に直結する**

### ## 承認ポイントを守る重要性
- 選定 → 計画 → 実装の各ステップで人間が承認を挟む
- 「計画を先に提示して承認後に実装」ルール（CLAUDE.md）が機能している実例
- 教訓②：**委任しても「選定の承認」だけは人間が握る**

### ## この記事で修正したバグの顛末
- Issue #14 の修正は4つの設計判断に連鎖した（詳細は別記事）
- ここでは「Issues起点で始めたからこそ問題の背景が把握しやすかった」点に言及
- 教訓③：**GitHub Issues に再現手順・期待動作が書いてあると、委任の精度が上がる**

### ## まとめ
- 3つの教訓を再掲
- 「委任範囲をどこまで広げられるか」の試行が発注スキルの鍛錬になる
