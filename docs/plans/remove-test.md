# 店候補機能 削除作業 指示書

## 目的

幹事ツールから店候補機能を完全に削除する。
スコープを「日程調整 + アレルギー対応」のみに絞る。

## 削除作業の前提ルール

- **完全削除すること。コメントアウトでの削除は禁止。**
- **削除作業のみを行うこと。新しい機能追加や既存機能の仕様変更は一切行わないこと。**
- **作業はブランチ `feature/remove-store-management` 上で実施すること。**
- **マイグレーションファイルを作成し、`stores`テーブル(該当テーブル)をdropすること。**
- **関連するimport文も全て削除すること。**

## 削除対象リスト

### フロントエンド

- [ ] 管理画面の「店候補追加」フォームコンポーネント
- [ ] 管理画面の「店候補一覧」表示コンポーネント
- [ ] 参加者画面の「店候補閲覧」セクション
- [ ] 店候補関連の型定義(types/store.ts 等)
- [ ] 店候補関連のAPIクライアント関数
- [ ] 店候補関連のルーティング(該当する場合)
- [ ] 店候補関連のテストファイル(該当する場合)
- [ ] 店候補関連のスタイル定義(該当する場合)

### バックエンド

- [ ] 店候補CRUD APIエンドポイント(POST/GET/PUT/DELETE)
- [ ] 店候補関連のzod schema
- [ ] 店候補関連のサービスロジック
- [ ] 店候補関連のDBクエリ(Drizzle)

### データベース

- [ ] storesテーブル(または相当するテーブル)のdropマイグレーション
- [ ] 関連する外部キー制約の削除
- [ ] Drizzle schema定義の削除

### ドキュメント

- [ ] README の機能説明から店候補関連を削除
- [ ] (利用規約があれば)店候補関連条項を削除

## 完了条件

- 提供した削除対象リストの全項目が処理されている
- `tsc --noEmit` 完走、型エラーゼロ
- ビルド完走
- linter完走、警告ゼロ
- 既存の日程調整機能・アレルギー機能が正常動作する
- マイグレーション適用後、storesテーブルが存在しない

## 完了報告に含めること

1. 削除したファイルのリスト
2. 修正したファイルのリスト(削除対象ではないが影響を受けたもの)
3. 作成したマイグレーションファイル名
4. 動作確認した内容
5. 残存懸念事項(あれば)

## 禁止事項

- 削除と関係のないリファクタリング
- 削除と関係のない依存関係の変更
- コメントアウトによる「擬似削除」
- 完了報告の捏造(実際には残っているのに「削除した」と報告すること)
