## 見出し構成と要点

### ## はじめに
- 「← トップに戻る」が常にトップに戻るというシンプルなバグを直そうとした
- React Router の location.state を使えばすぐ終わると思っていた
- 実際には4つの設計判断が次々と出てきた

### ## バグの概要：なぜ常にトップに戻るのか
- `<Link to="/">← トップに戻る</Link>` がハードコードされていた
- 管理ページ・イベントページ経由でも常に `/` に飛ぶ
- 3ページ（利用規約・プライバシーポリシー・FAQ）共通の問題
- **初回実装（Claude Code 完全委任）では気づかなかった**：法的ページは「作って終わり」で遷移元を意識していなかった。これが委任開発における「理解負債」の典型例

### ## 設計判断①：document.referrer か location.state か
- フッターに `<a target="_blank">` と `<Link>` の2種類が混在していた
- `target="_blank"` なら location.state は引き継げない → document.referrer で取得可能
- しかし SPA で `<Link>` を使った場合、document.referrer は空になる
- **教訓：`document.referrer` は新しいタブ遷移には使えるが SPA 遷移では使えない**

### ## 設計判断②：フッターリンクを \<Link\> に統一する
- SPA 内のページに `<a href target="_blank">` を使うのは慣習に反する
- `<Link state={{ from: location.pathname }}>` に統一することで実装が一本化できる
- `useLocation()` で `location.state?.from ?? '/'` を読めば完結
- **教訓：SPA 内リンクは外部リンクでない限り `<Link>` を使う**

### ## 設計判断③：「行き止まりページ」のフッターを消す
- 統一後も問題が残る：利用規約→プライバシーポリシー→FAQと移動するとfromが上書きされる
- 一般的なサービス（GitHub・Stripe等）では法的ページにナビゲーションフッターを置かない
- 「行き止まりページ」からフッターを消すことでチェーン遷移問題を根本解決
- **教訓：ユーザーが「読んで戻る」だけのページにはナビゲーションを置かない**

### ## 設計判断④：App.tsx から Footer を外してページ管理に移す
- 同一ルート `/event/:shareToken` で「フォーム表示」と「サンクス表示」の Footer を出し分ける必要が出た
- App.tsx のルートベース除外では同一ルートの state 差異を見分けられない
- Footer を各ページに移すことで、サンクス state のときだけ非表示にできる
- **教訓：グローバルレイアウトではサブ状態を制御できない。ページ固有のロジックはページに持たせる**
